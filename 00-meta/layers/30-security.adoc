= 30 Security

== Path
`domains/<domain>/30-layer-security/`

== Purpose
Слой хранит обязательные security constraints, baseline controls и security assessment влияния инициатив.

== Activation
- Этот слой входит в канонический стек `01-33`.
- Слой является самостоятельным canonical layer даже при отсутствии новых гипотез.

== Typical objects
- `SEC-*`

== Special fields
- `record_mode`: `baseline_constraint`, `initiative_assessment`, `mixed`
- `hard_constraints_text`
- `context_text`
- `assessment_subject_refs`
- `affected_requirement_refs`
- `security_scope`
- `data_classification`
- `threat_model_required`
- `security_decision`: `within_constraints`, `changes_required`, `constraint_violation`, `unknown`, `not_applicable`

== Traceability rules
- Baseline `SEC-*` может существовать без `HYP-*`, если фиксирует постоянный product constraint.
- Initiative-specific `SEC-*` должен быть связан хотя бы с одной `HYP-*`, `CHANGE-*` или другим assessment subject.
- Каждый `SEC-*` с решением `unknown`, `changes_required`, `constraint_violation` или неполным mandatory input должен иметь хотя бы один `RISK-*`, `REQ-*` или `CHANGE-*`.

== Key rules
- Для новых интеграций, API, данных, пользовательских сценариев и инфраструктурных изменений security assessment обязателен.
- Неполный security input оформляется через `mandatory_input_status`, `refusal_reasons` и `resulting_risk_refs`.
- Отказ от заполнения не является `not_applicable`.

== Related layers
- Upstream: `26 Business Hypotheses`, `22 Changes Roadmap`, `11 API Interfaces`, `12 Integrations`, `13 Data Architecture`, `07 Scenarios UX`.
- Downstream: `08 Requirements`, `18 Risks`, `22 Changes Roadmap`, `25 Traceability`.

== Authoring notes
- Карточка должна явно разделять две части: `Жесткие требования и ограничения` и `Контекст и оценка влияния`.
- Если `threat_model_required` фактически нужен, отсутствие связанных security inputs или артефактов должно быть зафиксировано как gap с риском, а не молчаливо пропущено.
- `not_applicable` допустим только при явном объяснении неприменимости.
- Baseline constraint cards могут переиспользоваться многими инициативами, а initiative-specific assessment cards могут ссылаться на существующие baseline cards вместо дублирования их содержимого.
