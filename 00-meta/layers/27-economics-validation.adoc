= 27 Economics Validation

== Path
`domains/<domain>/27-layer-economics-validation/`

== Purpose
Слой хранит постоянные budget and cost constraints и economic impact assessment инициатив.

== Activation
- Этот слой входит в канонический стек `01-33`.
- Слой является самостоятельным canonical layer даже при отсутствии новых гипотез.

== Typical objects
- `ECON-*`

== Special fields
- `record_mode`: `baseline_constraint`, `initiative_assessment`, `mixed`
- `hard_constraints_text`
- `context_text`
- `assessment_subject_refs`
- `affected_requirement_refs`
- `economic_driver`
- `cost_model`
- `expected_benefit`
- `budget_limit`
- `economic_decision`: `within_constraints`, `changes_required`, `constraint_violation`, `unknown`, `not_applicable`

== Traceability rules
- Baseline `ECON-*` может существовать без `HYP-*`, если фиксирует постоянный product constraint.
- Initiative-specific `ECON-*` должен быть связан хотя бы с одной `HYP-*`, `CHANGE-*` или другим assessment subject.
- Каждый `ECON-*` с решением `unknown`, `changes_required`, `constraint_violation` или неполным mandatory input должен иметь хотя бы один `RISK-*`, `REQ-*` или `CHANGE-*`.

== Key rules
- Слой поддерживает и baseline constraints, и initiative-specific assessment.
- Если инициатива выходит за economic constraints, это должно привести к `REQ-*`, `CHANGE-*`, `TASK-*`, `RISK-*` или к явному отказу.
- Ни один из слоев `27-33` не считается молча пройденным по умолчанию.
- Отказ от заполнения не является `not_applicable`: причина и связанный риск фиксируются явно.

== Related layers
- Upstream: `26 Business Hypotheses`, `22 Changes Roadmap`.
- Downstream: `08 Requirements`, `18 Risks`, `22 Changes Roadmap`, `23 Development Tasks`, `25 Traceability`.

== Authoring notes
- Карточка должна явно разделять две части: `Жесткие требования и ограничения` и `Контекст и оценка влияния`.
- Если карточка фиксирует только baseline constraints, в контекстной части нужно явно записать, что initiative-specific assessment отсутствует или не рассматривается.
- Если карточка фиксирует initiative-specific assessment, она должна ссылаться на существующие hard constraints или явно фиксировать создание нового constraint.
- `not_applicable` допустим только при явном объяснении неприменимости.
- Baseline constraint cards могут переиспользоваться многими инициативами, а initiative-specific assessment cards могут ссылаться на существующие baseline cards вместо дублирования их содержимого.
