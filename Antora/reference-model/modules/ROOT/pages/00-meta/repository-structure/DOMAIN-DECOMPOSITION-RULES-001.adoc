= DOMAIN-DECOMPOSITION-RULES-001

Канонические правила разбиения репозитория на домены и наследования локального `.summary` по уровням.

== 1. Назначение
- зафиксировать, что верхний структурный контейнер репозитория называется `domains/`
- определить правила выделения верхнеуровневых и дочерних доменов
- зафиксировать обязательное наследование root `00-meta/` и статического layer-reference из `00-meta/layers/`

== 2. Центр правды
- root `00-meta/` является единственным источником repository-wide правил, схем и канонического набора слоев
- если требуется изменить набор слоев, naming, schema или structural rules, изменение сначала вносится в root `00-meta/`
- домен и дочерний домен не имеют права самостоятельно вводить новый слой или отменять существующий

== 3. Базовая структура
- верхнеуровневый домен живет в `domains/<domain>/`
- дочерний домен живет в `domains/<domain>/domains/<child-domain>/`
- для любого домена канонический предметный стек считается полным только при готовности работать с полным root-defined набором слоев, даже если часть из них остается `unused`
- `.summary/` домена хранит только локальный meta/context уровень внутри унаследованного канона

== 4. Верхнеуровневый и дочерний домен
- если домен расположен в `domains/<domain>/`, он считается верхнеуровневым
- если домен расположен в `domains/<parent-domain>/domains/<child-domain>/`, он считается дочерним доменом
- для верхнеуровневого product domain `01-layer-aeilus-value-streams/` считается foundational-by-default и должен быть явно оценен до закрытия первой волны анализа
- для доменной карты или другого канонического meta-объекта домена должен поддерживаться реквизит `parent_domain`
- во всех дочерних доменах `parent_domain` обязателен и должен совпадать с фактическим положением домена в дереве
- у каждого домена должен быть явный ответственный, зафиксированный в `MAP-*` через поле `owner`

== 5. Правила наследования
- любой домен всегда наследует правила root `00-meta/`
- любой дочерний домен наследует и root `00-meta/`, и правила родительского домена
- наследование не может быть отключено локально
- дочерний домен не может менять унаследованные structural rules, обязательные поля и канонический набор слоев
- домен может фиксировать только локальный контекст, локальные ссылки, локальный набор используемых объектов и факт неиспользования части слоев
- дочерний домен может не materialize локальный слой `01` только при явной фиксации inheritance существующего value contract или отсутствия independent actor, value outcome или value movement
- домен должен быть способен жить и меняться автономно в пределах своего scope без неявной зависимости от соседних доменов

== 6. Что считается допустимым локальным уточнением
- указать, какие слои в конкретном домене фактически используются
- добавить локальные источники, риски, требования, traceability-связи и другие локальные объекты внутри унаследованного канона
- зафиксировать локальную терминологию, если она не конфликтует с root-правилами
- указать связь домена с родителем и соседними доменами
- указать domain owner и локальный change cadence в `.summary/MAP-*`

== 7. Что запрещено на уровне домена
- создавать новый слой без изменения root `00-meta/`
- удалять или переименовывать канонический слой
- менять смысл существующего слоя локальным решением домена
- переопределять обязательные поля карточек так, чтобы нарушать root schema contract
- подменять repository-wide governance локальным `.summary/`
- молчаливо закрывать первую волну анализа верхнеуровневого product domain без явной оценки слоя `01` или без omission note с причиной, source basis и linked `RISK-*`
- запускать cross-domain change без явной проверки impact и без cascade scope в реально затронутых доменах

== 8. Правила выделения домена
- домен выделяется по смысловой границе, а не только по количеству файлов
- у домена должна быть собственная ответственность, терминология, локальные решения или самостоятельный change cadence
- если область отличается только объемом файлов, но не имеет собственной семантики, новый домен создавать не нужно
- общие объекты, нужные нескольким доменам, поднимаются в `common/` или на уровень общего родителя

== 9. Общие объекты и `REF-*`
- общие объекты не копируются вниз без необходимости
- локальный reference-object создается только если прямой связи недостаточно для локальной навигации
- если достаточно прямой ссылки на объект выше по иерархии, дублирование запрещено

== 10. Запуск wave по доменам
- Wave начинается в том домене, к которому относится исходный signal.
- До расширения scope нужно проверить impact на root, parent и sibling domains.
- Подтвержденный impact оформляется как каскадная wave в затронутом домене с собственными объектами и traceability.
- Отсутствие confirmed cascade при cross-domain impact считается неполной formalization.

== 11. Порядок изменения канона
1. Обновить root `00-meta/`.
2. Синхронизировать `00-meta/layers/` и `00-meta/layers-summary/`.
3. Синхронизировать schemas и structural rules.
4. Применить изменение к доменам, которые используют затронутые слои.
5. Проверить traceability и совместимость доменов с новым состоянием канона.
