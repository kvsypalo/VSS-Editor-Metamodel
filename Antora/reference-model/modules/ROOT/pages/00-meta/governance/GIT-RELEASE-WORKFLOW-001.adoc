= GIT-RELEASE-WORKFLOW-001

Короткий корпоративный регламент для изменений по каноническому набору слоев модели.

== 1. Назначение
- Этот регламент задает единые правила работы с Git, ветками, approvals, release и traceability для любых изменений по каноническому набору слоев.
- Цель: сохранять честную инженерную историю, не терять связь между требованием, апрувом, реализацией и проверкой.

== 2. Область действия
- Регламент обязателен для всех изменений, которые затрагивают код, требования, сценарии, API, тесты, roadmap, traceability и другие слои модели.
- Регламент действует для сигналов от стейкхолдеров, product owner, аналитиков, разработки, тестирования, поддержки, архитектуры и эксплуатации.

== 3. Базовые принципы
- Изменение ведется волной, а не отдельным файлом.
- Одна волна покрывает один управляемый смысл: один change scope, один approval scope, один release scope.
- Product wave стартует в домене-источнике, а не сразу во всей метамодели.
- Перед расширением scope по всей модели wave обязана проверить impact на root, родительские и соседние домены и каскадироваться только в подтвержденно затронутые домены.
- История после публикации не переписывается. `push --force` запрещен, кроме отдельно согласованных аварийных случаев.
- В `main` попадают только изменения, прошедшие формализацию, approval, implementation и verification.
- Если процесс ранее был нарушен, история восстанавливается backfill-коммитами, а не переписыванием прошлого.
- AI-агент, работающий с репозиторием, обязан отслеживать этот регламент, проверять соответствие последовательности коммитов этапам wave и выполнять только допустимые по регламенту действия.
- AI-агент не имеет права сам нарушать этот регламент, а также не должен позволять нарушать его без отдельного явного подтверждения пользователя.

== 4. Обязательная цепочка change wave
Для каждой волны должны быть различимы следующие этапы:

1. `signal` - поступил входящий запрос или проблема.
2. `formalize` - изменение оформлено в карточках нужных слоев.
3. `approve` - scope официально согласован.
4. `implement` - выполнены изменения кода и нижних слоев.
5. `verify` - есть test evidence и подтверждение результата.
6. `trace sync` - обновлены связи между слоями.
7. `release` - изменение попало в `main` и при необходимости отмечено тегом.

== 5. Обязательные правила Git
- Прямая работа в `main` запрещена.
- Каждая волна идет через отдельную ветку от актуального `main`.
- В одной ветке нельзя смешивать независимые инициативы.
- Approval-коммиты нельзя смешивать с implementation-коммитами.
- Verification-коммиты нельзя смешивать с implementation-коммитами.
- До явного approval допускается только formalize-коммит. Коммиты `Implement`, `Add test evidence`, `Verify and sync`, `Sync traceability` и любые release-действия до approval запрещены.
- Merge в `main` выполняется через Pull Request.
- Если важна честная история, использовать `Create a merge commit`, не использовать squash.
- Коммиты должны отражать этап процесса, а не факт редактирования файлов.

== 6. Обязательные правила по каноническому набору слоев
- Если меняется продуктовый контракт, должны обновляться соответствующие карточки верхних и нижних слоев.
- Любая product wave, меняющая actor-facing behavior, обязана зафиксировать в слое `01` изменение ценности и anti-value; если инициатор этого не описал, AI-агент обязан явно спросить об этом до formalize-stage.
- Если изменение реализовано в коде, должны быть синхронизированы как минимум requirements, tasks, tests и traceability.
- Нельзя ставить `tested_passed` без фактической проверки.
- Нельзя считать волну завершенной, если связи между слоями не обновлены.
- Traceability обновляется в той же волне, где изменение подтверждено.

== 7. Типы веток
- `feature/<topic>` - новая функциональность.
- `change/<topic>` - волна изменений с акцентом на модель и согласование.
- `hotfix/<topic>` - срочное исправление для продовой проблемы.
- `release/<version>` - предрелизная стабилизация.

== 8. Рекомендуемая структура коммитов
Минимально рекомендуемая последовательность:

1. `Add <scope> requirements`
2. `Approve <scope>`
3. `Implement <scope>`
4. `Add <scope> test evidence`
5. `Sync <scope> traceability`

Правила интерпретации этой последовательности:

- `Add <scope> requirements` - единственный допустимый коммит до approval.
- Без явного `Approve <scope>` никакие downstream-коммиты не выполняются: ни `Implement`, ни `Add test evidence`, ни `Verify and sync`, ни `Sync traceability`, ни release-коммиты.
- После `Approve <scope>` обязательна полная последовательность downstream-коммитов без пропуска этапов: `Implement`, затем `Add <scope> test evidence`, затем `Sync <scope> traceability`.
- Если verification и traceability действительно не разделяются, вместо двух последних коммитов допускается один объединенный `Verify and sync <scope>`.
- AI-агент обязан контролировать эту последовательность, явно указывать на нарушение и не продолжать workflow без отдельного подтверждения пользователя.

Если verification и model sync не разделяются, допустим объединенный коммит:

1. `Verify and sync <scope>`

== 9. Типовые входы по ролям
- Стейкхолдер: формирует бизнес-сигнал, который переводится в goals, value, constraints и requirements. В код переходить только после product approval.
- Product Owner: формирует scope поставки, acceptance criteria и release intent. После approval волна идет в реализацию.
- Для product owner и аналитика отсутствие явного value/anti-value delta считается недоформализованным signal и закрывается отдельным вопросом до downstream-этапов.
- Аналитик: формирует findings, уточнения требований, сценариев, API и traceability. После согласования идет implementation.
- Разработка: формирует technical change, refactoring, debt reduction или engineering constraint. Если меняется внешнее поведение, требуется product или engineering approval до кода.
- Тестирование: формирует defect, regression или failed validation. После приоритизации идет fix wave, затем re-test и trace sync.
- Поддержка: формирует incident, user pain или эксплуатационную проблему. Для срочного случая создается `hotfix`, для системного случая - обычная change wave.
- Архитектура и эксплуатация: формируют structural changes, NFR, KPI, observability-driven findings. Изменения проходят ту же цепочку formalize -> approve -> implement -> verify.

== 9a. Правила domain-origin cascade
- Каждая wave начинается в том домене, к которому относится исходный signal.
- До approval и implementation нужно проверить, не затрагивает ли изменение root-scope, родительский домен или соседние домены через общие процессы, требования, traceability или inherited contracts.
- Если impact подтвержден, в затронутом домене запускается каскадная wave с собственным scope, объектами и traceability, но связанная с исходной wave.
- Отсутствие cascade при подтвержденном cross-domain impact считается незавершенной formalization.

== 10. Правила merge и release
- PR в `main` означает запрос на включение подготовленной волны в основную ветку продукта.
- Перед merge должна быть проверена полная цепочка: requirement -> approval -> implementation -> verification.
- Для сохранения честной истории использовать merge commit без squash.
- После merge в `main` при необходимости создается релизный тег.
- Если используется `release/<version>`, в нее допускаются только стабилизация, release notes и безопасные bugfix.
- Hotfix после выпуска обязан быть возвращен обратно в активные ветки, если они существуют.

== 11. Что запрещено
- Смешивать несколько независимых change waves в одной ветке.
- Делать approval задним числом внутри implementation-коммита.
- Делать любой downstream-коммит до явного approval, кроме formalize-коммита.
- Пропускать обязательные пост-approval этапы `Implement -> Add test evidence -> Sync traceability` или допустимый объединенный этап `Verify and sync`.
- Делать verification без реализации.
- Мержить в `main` волну без test evidence и trace sync.
- Переписывать опубликованную историю там, где проблему можно решить честным backfill.

== 12. Критерий завершения волны
Волна считается завершенной, если:

1. есть формализованный scope изменения;
2. есть явный approval;
3. реализованы код и затронутые слои;
4. есть test evidence;
5. синхронизирована traceability;
6. изменение влито в `main` и при необходимости выпущено тегом.
