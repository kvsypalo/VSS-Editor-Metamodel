= ARTIFACT-NOTATION-GOVERNANCE-001

Корпоративный регламент по допустимым нотациям, внешнему виду и обязательной идентификации артефактов.

== 1. Назначение
- Этот регламент задает контролируемый whitelist нотаций для артефактов репозитория.
- Цель: не допускать произвольных diagram styles от AI-агентов и людей, удерживать семантическую корректность нотаций и опираться на современные практики моделирования, используемые в ИТ-индустрии.

== 2. Область действия
- Регламент обязателен для всех файлов в `artifacts/`, которые используются как исходники диаграмм, карт, схем и визуальных моделей.
- Регламент обязателен для карточек, которые создают, изменяют, ссылаются на или валидируют такие артефакты.
- Растровые экспортные файлы `png`, `jpg`, `jpeg`, `webp`, `gif` не считаются каноническим исходником нотации и допустимы только как preview/export рядом с исходным source-артефактом.

== 3. Базовые принципы
- Каждый артефакт обязан явно указывать используемую нотацию.
- В одном артефакте допускается только одна основная нотация. Смешивание BPMN, EPC, UML, C4, ArchiMate и других нотаций на одном полотне запрещено.
- Инструмент, редактор или AI-агент, в котором создан артефакт, не определяет правила нотации. Он фиксирует происхождение артефакта, но не ослабляет требования к корректности нотации.
- Внешний вид должен следовать стандартным графическим примитивам выбранной нотации и не должен заменяться произвольными иллюстрациями, декоративными иконками, градиентами, тенями и случайной цветографикой.
- Если для задачи нужна другая визуальная модель, сначала выбирается нотация из whitelist. Создание пользовательской нотации допускается только отдельным governance-решением.
- Каноническим считается редактируемый source-артефакт, а не его экспорт в картинку.

== 4. Обязательная идентификация нотации
Каждый канонический артефакт обязан содержать минимум следующие поля идентификации:

- `notation` - имя нотации из whitelist
- `style_source` - источник отрисовки или authoring environment, в котором создан артефакт, например `Business Studio`, `draw.io`, `PlantUML`, `Mermaid`, `Visio`, `GitHub Copilot`, `manual SVG`
- `viewpoint` - тип представления внутри нотации, если применимо
- `source_format` - формат исходного файла

Поле `style_source` нужно для traceability происхождения артефакта. Нормативным источником правил остается сама нотация, а не инструмент, в котором артефакт был создан.

Предпочтительный способ фиксации:

1. внутри самого source-файла в машиночитаемом metadata/comment/header блоке;
2. если формат поддерживает XML metadata, использовать XML-комментарий или `<metadata>`;
3. для текстовых DSL использовать comment-header в начале файла;
4. экспортные raster preview не могут быть единственным носителем нотации.

== 5. Разрешенный whitelist нотаций
Ниже перечислены нотации и типы схем, которые разрешены для этого репозитория как современные, практически применимые в ИТ-индустрии и поддержанные внутренней экспертной позицией аналитиков и руководства.

|===
| Семейство | Нотация | Где использовать | Комментарий по применению 
| Процессы | `IDEF0` | контекстные и функциональные модели верхнего уровня | Использовать ICOM-логику: вход, управление, выход, механизм 
| Процессы | `VAD` | value-added chain, value stream, сквозные цепочки | Использовать для верхнеуровневых потоков ценности и этапов 
| Процессы | `FAD` | функционально-распределенные схемы | Использовать, когда важно распределение функций по участникам или объектам 
| Процессы | `Basic Flowchart` | простые процедуры и последовательности действий | Использовать для линейных и простых ветвящихся сценариев 
| Процессы | `Cross-functional Flowchart` | процедуры с зонами ответственности | Использовать, когда основной смысл в передаче работы между ролями/подразделениями 
| Процессы | `BPMN 2.0` | исполнимые и аналитические process flows | Использовать задачи, события, шлюзы, пулы и дорожки без подмены символов 
| Процессы | `EPC` | событийно-функциональные цепочки процессов | Использовать чередование событий и функций, операторы AND/OR/XOR 
| Стратегия | `Strategic map` | карты целей и причинно-следственных связей | Использовать для целей, KPI и связей достижения без смешивания с process-flow символами 
| Архитектура | `ArchiMate` | enterprise architecture viewpoints | Использовать стандартные слои и аспекты языка ArchiMate 
| Архитектура | `C4` | context/container/component/code views | Указывать конкретный уровень C4 в `viewpoint` 
| Архитектура | `UML` | структурные и поведенческие диаграммы систем | Указывать конкретный вид диаграммы в `viewpoint` 
| Данные | `ERD` | модели сущностей и связей | Использовать единообразное обозначение сущностей, атрибутов и кардинальностей 
| Данные/Интеграция | `DFD` | потоки данных и границы обработки | Использовать процессы, хранилища данных, внешние сущности и потоки 
|===

== 6. Обязательные правила внешнего вида
- Фигуры должны быть нативными для выбранной нотации. Нельзя рисовать BPMN-задачи как произвольные карточки, EPC-функции как UML-классы, C4-контейнеры как flowchart-ромбы и т.д.
- Цвет должен быть сдержанным и вторичным по отношению к форме. Базовый режим: светлый фон, читаемые подписи, темный контур, минимальное количество акцентных цветов.
- Подписи должны быть функциональными и короткими: действия формулировать глаголом, сущности и объекты именовать существительными.
- Связи и стрелки должны соответствовать правилам выбранной нотации, а не дизайнерским предпочтениям автора.
- Если визуализация перегружена, ее надо декомпозировать на несколько связанных артефактов, а не вводить новые условные обозначения.

== 7. Минимальные правила по отдельным нотациям
- `IDEF0`: использовать контекстную диаграмму `A-0` для верхнего уровня и сохранять стороны ICOM без перестановки смысла.
- `VAD`: использовать последовательность этапов value stream без смешивания со шлюзами BPMN и без детальной исполнимой логики. Для внешнего референса по value stream/value system visual style можно использовать пример VSS-editor: https://aeilus.tech/vss-editor/?schema=https://aeilus.tech/vss-editor/json/pmbok2.json . Конкретный машинный формат принятия таких артефактов должен быть утвержден отдельным change scope.
- `FAD`: выделять распределение функций по участникам, системам или объектам, а не подменять FAD обычным flowchart.
- `Basic Flowchart`: применять для простого procedural flow без BPMN-пулов и EPC-событий.
- `Cross-functional Flowchart`: основной смысл должен читаться через lanes/roles и переходы между ними.
- `BPMN 2.0`: использовать задачи/подпроцессы, события, шлюзы, message/sequence flow, пулы и дорожки по правилам BPMN.
- `EPC`: выдерживать логику событий и функций; не заменять события BPMN-иконками.
- `Strategic map`: использовать цели, показатели и причинно-следственные связи без BPMN-шлюзов, UML-стрелок и произвольных карточных виджетов.
- `ArchiMate`: не смешивать элементы разных viewpoint без явного архитектурного обоснования.
- `C4`: обязательно указывать один из уровней `context`, `container`, `component`, `code`.
- `UML`: обязательно указывать вид диаграммы, например `use-case`, `class`, `sequence`, `activity`, `state`.
- `ERD`: в одном артефакте использовать только один способ записи кардинальностей.
- `DFD`: не смешивать DFD с BPMN или UML activity на одном полотне.

== 8. Примеры допустимой идентификации

=== 8.1 BPMN XML
[source,xml]
----
<!--
notation: BPMN 2.0
style_source: Camunda Modeler
viewpoint: process-flow
source_format: bpmn+xml
-->
<bpmn:definitions ...>
----

=== 8.2 SVG для VAD или стратегической карты
[source,xml]
----
<!--
notation: VAD
style_source: draw.io
viewpoint: value-stream
source_format: image/svg+xml
-->
<svg ...>
  <metadata>notation=VAD;style_source=draw.io;viewpoint=value-stream</metadata>
----

=== 8.3 PlantUML для UML
[source,text]
----
' notation: UML
' style_source: PlantUML
' viewpoint: sequence
' source_format: plantuml
@startuml
----

=== 8.4 Mermaid для C4
[source,text]
----
%% notation: C4
%% style_source: Mermaid
%% viewpoint: container
%% source_format: mermaid
----

== 9. Примеры выбора нотации
- `ART-XXX-order-to-cash-context. svg` -> `notation: VAD`, если нужен верхнеуровневый поток создания ценности.
- `ART-XXX-order-to-cash-execution.bpmn` -> `notation: BPMN 2.0`, если нужен детальный исполнимый или аналитический flow.
- `ART-XXX-company-goals.svg` -> `notation: Strategic map`, если нужен обзор целей, KPI и причинно-следственных связей.
- `ART-XXX-application-landscape.svg` -> `notation: ArchiMate`, если нужен enterprise architecture viewpoint.
- `ART-XXX-integration-context.mmd` -> `notation: C4`, если нужен system context или container view.
- `ART-XXX-domain-model.puml` -> `notation: UML`, `viewpoint: class`, если нужен доменный или технический class model.
- `ART-XXX-customer-data-model.drawio` -> `notation: ERD`, если нужна модель сущностей и связей.
- `ART-XXX-payment-data-flow.svg` -> `notation: DFD`, если нужен поток данных между внешними сущностями, процессами и хранилищами.

== 10. Что запрещено
- Рисовать "авторскую нотацию", если она не утверждена отдельным governance-документом.
- Смешивать несколько нотаций в одном артефакте для экономии места.
- Оставлять артефакт без явного указания `notation`.
- Хранить только raster-export без редактируемого source-файла.
- Подменять правила нотации визуально красивыми, но семантически неправильными элементами.

== 11. Источники governance
- Общепринятые отраслевые правила и спецификации нотаций: BPMN 2.0, UML, ArchiMate, IDEF0 и связанные с ними практики моделирования.
- Современные архитектурные и аналитические практики ИТ-индустрии: C4, ERD, DFD, process flow, strategy map, value stream mapping.
- Практический опыт аналитиков, архитекторов и руководства, участвующих в развитии репозитория и принимающих решения по качеству моделей.
- Внутренний governance репозитория, который ограничивает допустимый набор нотаций и запрещает произвольные пользовательские обозначения без отдельного решения.
